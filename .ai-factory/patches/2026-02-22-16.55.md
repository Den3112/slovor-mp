# CSP Blocking Supabase Client Connections

**Date:** 2026-02-22 16:55
**Files:** proxy.ts
**Severity:** high

## Problem

TypeError: Failed to fetch displaying in the console with empty `{}` payloads (until logger improvement) or generic "Failed to fetch". This occurred specifically on client-side Supabase requests (like `messagesApi.getUnreadCount`).

## Root Cause

A Content Security Policy (CSP) was implemented in `proxy.ts`, but it lacked a `connect-src` directive. According to CSP rules, if `connect-src` is missing, browsers fall back to applying `default-src` to API/fetch connections. Since `default-src` was strictly set to `'self'`, all cross-origin XHR, Fetch, and WebSocket traffic was blocked by the browser. This prevented the client-side Next.js code from reaching `*.supabase.co`.

## Solution

Added an explicit `connect-src` directive to the CSP header in `proxy.ts`:
`connect-src 'self' https://*.supabase.co wss://*.supabase.co https://api.exchangerate-api.com https://api.stripe.com;`
This allows standard Fetch and WebSocket communication with external dependencies while maintaining CSP protection against injection.

## Prevention

- When implementing a Strict CSP, always verify `connect-src` for any third-party services utilized by client-side code (Supabase, Firebase, Stripe, Analytics, etc.).
- Never depend only on `.next/server` logs because CSP drops requests natively in the browser before they hit the network. Console inspections are vital.
