# Missing Category Icon Selection and Mapping Inconsistency

**Date:** 2026-02-23 10:49
**Files:** lib/constants/category-icons.ts, components/features/dashboard/admin/categories-view/index.tsx, components/features/dashboard/admin/categories-view/category-form-dialog.tsx
**Severity:** medium

## Problem

Admins could not select Lucide-based icons (`icon_name`) from the Categories management page, only emojis. Additionally, the admin panel used a very limited and disconnected `iconMap`, causing inconsistent icon displays between the admin panel and the public site.

## Root Cause

1.  The `icon_name` field exists in the database/schema but was omitted from the admin the `CategoryFormDialog`.
2.  `AdminCategoriesView` had a hardcoded `iconMap` that didn't sync with the global `CATEGORY_ICON_MAP` used in public sections.
3.  The rendering logic prioritized slug-based lookup over explicitly set database values.

## Solution

1.  **Unified Icon Registry**: Updated `lib/constants/category-icons.ts` to export `ICON_NAME_MAP` and `AVAILABLE_ICON_NAMES`, creating a single source of truth for both public and admin views.
2.  **Advanced Form Picker**: Enhanced `CategoryFormDialog` with a Lucide icon selector (select dropdown), an emoji input, and a live preview area. Added "Quick Select" buttons for common icons.
3.  **Robust Rendering**: Refactored `CategoryIcon` (both local and global) to use a tiered priority system: explicit emoji > explicit Lucide name > slug-based global default.
4.  **Logging**: Added `[FIX]` logging to the category submission handler to verify payloads.

## Prevention

- Ensure all database schema fields (`icon_name` in this case) are accounted for in admin management forms.
- Centralize data structures like icon maps and theme constants in `lib/constants` instead of local component states.
- Always provide a preview when editing visual attributes (icons, colors) to reduce admin error.
