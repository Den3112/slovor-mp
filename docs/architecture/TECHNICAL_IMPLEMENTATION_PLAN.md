# Технический план реализации ключевых функций Slovor Marketplace

Этот документ описывает конкретный технический план для превращения Slovor Marketplace в лучший маркетплейс объявлений в Словакии. Он опирается на текущую архитектуру (Next.js App Router, Supabase, модульный API-слой, i18n) и дополняет существующие документы как основной план развития.

## 0. Базовые принципы

* Использовать существующий API-слой `lib/api/*`, а не прямые обращения к Supabase в компонентах.
* Следовать 8 принципам из AI_GUIDE (малые компоненты, явная обработка ошибок, минимальный глобальный стейт).
* Новые UI-элементы строить поверх текущего UI-кита в `components/ui/*` и существующих доменных компонентов.
* Каждую крупную фичу сопровождать хотя бы базовыми unit-тестами (Vitest) и по возможности E2E-сценарием (Playwright).

## 1. Завершение критических флоу

### 1.1. Создание и редактирование объявлений

**Цель:** надёжный трёхшаговый флоу создания/редактирования объявления без потерь данных.

#### 1.1.1. Валидация формы

* Файл: `components/listing/create-listing-form.tsx`.
* Действия:
  1. Вынести схему формы в отдельный модуль, например `lib/utils/listing-form-schema.ts`:
     * Типизированная структура полей (title, price, category_id, images, location и т.д.).
     * Простая валидация (можно без внешней либы на первом этапе).
  2. В `CreateListingFormContent` вызывать валидацию перед переходом между шагами и перед `handleSubmit`.
  3. Отображать ошибки рядом с полями и агрегированно в верхнем алерте.

#### 1.1.2. Авто-сохранение черновика

* Новый модуль: `lib/utils/draft-storage.ts`.
* Идея:
  * Сохранять состояния формы в `localStorage` под ключом вида `listing-draft:${userId}`.
  * При монтировании `CreateListingFormContent` пытаться восстановить черновик (если нет `editId`).
* Технически:
  * В `useEffect` отслеживать изменения `formData` и с дебаунсом (300–500 мс) писать в `localStorage`.
  * При успешном сохранении объявления — очищать черновик.

#### 1.1.3. Предупреждение о несохранённых данных

* В `CreateListingFormContent` добавить обработчик `beforeunload` (только если есть несохранённые изменения):
  * В `useEffect` подписаться на событие, если форма модифицирована и не была успешно сохранена.
* Для роутера Next.js можно дополнительно сделать мягкое предупреждение при навигации (через собственный hook и диалог из UI-кита).

### 1.2. Поиск, фильтры и пустые состояния

**Цель:** стабильный поиск с хорошими UX-сценариями.

* Файлы: `app/listings/page.tsx`, `components/listing/filters.tsx`, `components/listing/ListingsView.tsx`.

#### 1.2.1. Унификация параметров поиска

* В `lib/api/listings.ts` уже есть `ListingFilterOptions`. Убедиться, что:
  * Все фильтры и сортировки, используемые в UI, строго отображаются на это API (priceMin, priceMax, condition, location, sort и т.д.).
  * В `app/listings/page.tsx` происходит корректный парсинг `searchParams` с дефолтами.

#### 1.2.2. Пустые состояния

* В `ListingsView` и/или странице списка добавить единый компонент пустого состояния:
  * Файл: `components/ui/empty-state.tsx` (уже есть заготовка — расширить и переиспользовать).
  * Текст: "Нет результатов" + кнопка сброса фильтров + ссылки на популярные категории.

#### 1.2.3. Тесты

* Unit:
  * Тесты для функций парсинга фильтров и `buildListingsQuery`.
* E2E:
  * Сценарий: пользователь открывает `/listings`, применяет фильтры, сбрасывает их, использует поиск.

## 2. Доверие и безопасность

### 2.1. Профиль пользователя и публичная страница продавца

**Цель:** дать покупателю контекст о продавце.

#### 2.1.1. Расширение схемы БД

* Таблица `profiles` (если ещё нет):
  * Поля: `id` (равен auth.user.id), `display_name`, `avatar_url`, `bio`, `created_at`, `updated_at`.
* Индексация по `id`.

#### 2.1.2. API-слой

* Новый модуль: `lib/api/profiles.ts`:
  * `getById(userId)`
  * `update(userId, updates)`

#### 2.1.3. UI

* Страница профиля пользователя: `app/profile/page.tsx`:

* Публичная страница продавца: `app/seller/[id]/page.tsx`:

### 2.2. Рейтинг и отзывы

#### 2.2.1. Схема БД

* Таблица `reviews`:
  * `id` (UUID)
  * `seller_id` (FK → users)
  * `buyer_id` (FK → users)
  * `listing_id` (FK → listings)
  * `rating` (int, 1–5)
  * `comment` (text, опционально)
  * `created_at`
* Индексы по `seller_id`, `listing_id`.

#### 2.2.2. API

* Новый модуль: `lib/api/reviews.ts`:
  * `create(review)` — создание отзыва (проверка, что buyer ≠ seller, и что покупатель участвовал в сделке — на первом этапе можно упростить).
  * `getForSeller(sellerId)` — список отзывов и средний рейтинг.

#### 2.2.3. UI

* Компонент: `components/profile/SellerRating.tsx`:
  * Показывает средний рейтинг, количество отзывов и несколько последних комментариев.
* На странице профиля и продавца — вывод этого компонента.

### 2.3. Жалобы и модерация

#### 2.3.1. Схема БД

* Таблица `reports`:
  * `id`
  * `reporter_id` (FK → users)
  * `reported_listing_id` (FK → listings, опционально)

#### 2.3.3. UI

* Кнопка "Report listing" на детальной странице объявления (уже есть перевод в `t.listing.reportListing`):
  * Открывает диалог (компонент на базе `components/ui/dialog.tsx`).
  * Позволяет выбрать причину и отправить жалобу.
* Простая админ-страница (можно только в dev/admin окружении): `app/admin/reports/page.tsx`.

## 3. Коммуникация: чат/сообщения

### 3.1. Схема БД

* Таблица `conversations`:
  * `id`
  * `listing_id`
  * `buyer_id`
  * `seller_id`
  * `created_at`
* Таблица `messages`:
  * `id`
  * `conversation_id`
  * `sender_id`
  * `content` (text)
  * `created_at`

### 3.2. API

* Новый модуль: `lib/api/messages.ts`:
  * `createConversation(listingId, buyerId, sellerId)`
  * `getOrCreateConversation(listingId, buyerId, sellerId)`
  * `getConversationsForUser(userId)`
  * `getMessages(conversationId)`
  * `sendMessage(conversationId, senderId, content)`

### 3.3. UI

* Список диалогов: `app/messages/page.tsx`:
  * Лист диалогов с последним сообщением, связанной карточкой объявления.
* Страница диалога: `app/messages/[id]/page.tsx`:
  * Лента сообщений, форма ввода.
* На странице объявления:
  * Кнопка "Написать продавцу" → создаёт/открывает диалог и редиректит в `/messages/[id]`.

### 3.4. Realtime (опционально во 2-й итерации)

* Использовать Supabase Realtime для live-обновлений сообщений.
* Подписка в клиенте по `conversation_id`.

## 4. Поиск и ранжирование

### 4.1. Ранжирование

* В `lib/api/listings.ts` расширить `ListingFilterOptions`:
  * `boostWithImages?: boolean`
  * `boostRecent?: boolean`
* На первом этапе:
  * В запросе сортировать по `views`, `created_at` и флагу `has_images` (вычисляется на основе `array_length(images, 1) > 0`).

### 4.2. Сигналы качества

* Добавить в таблицу `listings` поля:
  * `contact_clicks` (int)
  * `complaints_count` (int)
* В местах клика по телефону/контакту увеличивать `contact_clicks`.
* Использовать `complaints_count` и `contact_clicks` как дополнительные сигналы при сортировке.

### 4.3. Текстовый поиск

* На Supabase/Postgres стороне использовать `tsvector` для полнотекстового поиска по title+description (в будущем).
* Пока — улучшить текущий `ilike`-поиск, нормализуя запросы (lowercase, trim, возможно удаление диакритики).
